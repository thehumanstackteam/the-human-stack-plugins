# Expectation Mapper

The theoretical engine of Tim's practice. Takes any engagement context — a JTBD analysis, a call transcript, a client relationship — and maps the full expectation landscape. Every plugin (Strategist, Speechwriter, Instructinator) uses this framework, but this skill runs it explicitly and produces a standalone deliverable.

## Why This Exists

Tim's core thesis: **Emotion = Experience ± Expectations.** You can design the perfect experience and still fail if expectations are misaligned. Most practitioners try to improve the experience. Tim's edge is managing expectations — the cheaper, faster, more durable lever.

This skill makes that thesis operational. It answers three questions:
1. What expectations exist right now? (Origins)
2. How are they behaving? (Dynamics)
3. What should Tim do about them? (Architecture)

## The Framework: Expectation Architecture

Three layers. Each builds on the previous.

## Input Types

This skill accepts any of the following:

- **JTBD analysis** — Richest source. Maps directly to origins.
- **Call transcript** — Extract expectations from what was said and implied.
- **Client relationship summary** — Engagement history, touchpoints, artifacts delivered.
- **Proposal or SOW** — What was declared vs. what was left ambiguous.
- **Pre-session context** — "I'm meeting with [X] tomorrow, here's what I know."
- **Post-session debrief** — "Here's what happened, map what I think their expectations are now."

## How This Skill Connects to Plugins

### Strategist Plugin
- **strategy-audit** uses Layer 1 (Origins) to determine if Tim is meeting or missing expectations
- **strategy-architect** uses Layer 3 (Architecture) to build deliverables that close expectation gaps
- **strategy-evaluate** uses all three layers to assess post-delivery expectation alignment

### Speechwriter Plugin
- **speech-design** uses Layer 2 (Peak-End Weight) to design emotional arc and close
- **speech-audit** uses Layer 1 to check if the talk addresses what the audience expects
- **speech-evaluate** uses Layer 2 (Decay) to assess whether the talk's impact will persist

### Instructinator Plugin
- **session-architect** uses Layer 1 (Declared + Default) to design learning outcomes that match participant expectations
- **engagement-check** uses Layer 2 (Peak-End) to ensure interaction peaks are well-placed
- **dry-run-coach** uses Layer 3 (Monitor) to coach Tim on checking alignment during delivery

### Mode Router
- mode-router uses this skill's output to determine if the ENGAGEMENT expectations align with the MODE Tim is about to deliver in. Misalignment between engagement expectations and delivery mode is the #1 risk signal.

## Running This Skill From JTBD Data

When a JTBD analysis is available, the mapping is direct. Here's the extraction sequence:

1. **Pull section** → DECLARED expectations (what they want from Tim)
2. **Push section** → TRANSFERRED expectations (what was bad before, so they expect different)
3. **Anxiety section** → UNMANAGED expectations (fears that become invisible standards)
4. **Habit section** → DEFAULT expectations (current baseline everything gets compared to)
5. **Desired Outcome** → DECLARED expectations (success criteria, often vague)
6. **Social Job** → SOCIAL expectations (how stakeholders need to be perceived)
7. **Messaging Gold** → Language for DECLARE and INOCULATE operations
8. **Obstacles** → Expectations that need AUDIT (still unclear) or REDIRECT (already drifting)

## Theoretical Lineage

This framework synthesizes from established research to fill a gap none of them address:

- **Oliver (1980)** — Expectation Disconfirmation Theory: expectations as reference point for satisfaction. *We use:* the core equation (outcome vs. expectation).
- **Kahneman (1979)** — Prospect Theory: loss aversion, reference dependence. *We use:* Asymmetry dynamic (negative violations hurt 2x).
- **Parasuraman, Zeithaml, Berry (1985/1991)** — SERVQUAL, Zone of Tolerance: adequate vs. desired service levels. *We use:* the concept that expectations have a range, not a point.
- **Zeithaml, Berry, Parasuraman (1993)** — Antecedents of Expectations: explicit promises, implicit promises, word-of-mouth, past experience. *We use:* Origins layer (but expanded and renamed for operational use).
- **Kahneman et al. (1993)** — Peak-End Rule: memory of experiences dominated by peak and end. *We use:* Peak-End Weight dynamic.
- **Vroom (1964)** — Expectancy Theory: effort → performance → outcome chain. *We use:* the insight that expectations govern motivation to engage.

**The gap this framework fills:** None of the above provide a practitioner-level system for proactively designing and managing expectations in a service relationship. They describe what expectations do. This framework tells Tim what to do about them.

## What This Skill Does NOT Do

- Design deliverables (use strategy-architect)
- Detect mode drift in real time (use strategy-audit)
- Design talks or training sessions (use speech-design or session-architect)
- Replace JTBD analysis (use jtbd-analysis to discover expectations first, then feed to this skill)
- Make content decisions — this skill is about expectations, not content
